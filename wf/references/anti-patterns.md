# 工作流反模式

## 概述

反模式是看似合理但会导致问题的常见做法。识别并避免这些模式可以减少工作流卡顿、提高产出质量、降低 context 浪费。

## 执行反模式

### 1. 直接写入 STATE.md

**现象:** Agent 手动编辑 STATE.md 文件内容。
**问题:** 格式错误、字段遗漏、与 CLI 工具状态不一致。
**正确做法:** 始终通过 `wf-tools state` CLI 命令更新状态。CLI 工具会处理格式、边界情况和一致性。

### 2. 深层 Task() 嵌套

**现象:** Agent 在 Task() 内再次调用 Task()，形成多层嵌套。
**问题:** Context 消耗成倍增长，嵌套 agent 无法共享父级状态，错误难以追溯。
**正确做法:** 使用 Skill() 链替代嵌套 Task()。Skill() 在当前 context 内执行，无额外开销。

### 3. 跳过验证直接推进

**现象:** 执行完所有任务后直接标记阶段完成，不运行验证。
**问题:** 未发现的集成问题在后续阶段放大，修复成本翻倍。
**正确做法:** 每个阶段完成后必须运行 4 级验证（EXISTS → SUBSTANTIVE → WIRED → DATA-FLOWING）。

### 4. 忽略 Context 预算警告

**现象:** 收到 WARNING/CRITICAL 警告后继续执行复杂任务。
**问题:** Context 耗尽导致输出质量急剧下降，可能丢失未提交的工作。
**正确做法:** WARNING 时收尾当前任务，CRITICAL 时立即保存状态并暂停。参见 [context-budget.md](./context-budget.md)。

### 5. 一次性大批量文件修改

**现象:** 单个任务修改 10+ 个文件。
**问题:** 无法原子回滚，出错影响范围过大，commit 信息模糊。
**正确做法:** 每个任务聚焦 3-5 个相关文件，每完成一个任务立即 commit。

## 规划反模式

### 1. 超大计划（>3 任务/计划）

**现象:** 单个 PLAN.md 包含 5 个以上任务。
**问题:** 执行时间过长，context 不够用，中途失败浪费大量工作。
**正确做法:** 每个计划最多 3 个任务。超过时拆分为多个 wave 或多个计划文件。

### 2. 缺少依赖分析

**现象:** 多个计划之间没有声明 `depends_on`。
**问题:** 并行执行时出现冲突、编译错误或逻辑不一致。
**正确做法:** 在 PLAN.md frontmatter 中明确声明 `depends_on` 关系。无依赖的计划可并行，有依赖的必须顺序执行。

### 3. 模糊的任务描述

**现象:** 任务 action 写成 "实现认证功能" 或 "优化性能"。
**问题:** Executor agent 需要大量探索才能理解意图，浪费 context。
**正确做法:** 明确指定要修改的文件、函数签名、数据结构和验证方法。

## Agent 反模式

### 1. 缺少完成标记

**现象:** Agent 执行结束时没有返回结构化 JSON 完成标记。
**问题:** Orchestrator 无法判断成功/失败，默认视为失败并触发不必要的重试。
**正确做法:** 始终返回 `{ "status": "complete|partial|failed", "artifacts": [...], "summary": "..." }`。

### 2. 忽视 Context 预算

**现象:** Agent 在 context 使用率超过 70% 后仍继续执行新任务。
**问题:** 输出质量下降，可能在任务中途耗尽 context。
**正确做法:** 每个任务间隙检查 context 指标，超过 70% 时生成 partial SUMMARY 并优雅退出。

### 3. Agent 自行决定架构变更

**现象:** Executor 发现需要新数据库表或新服务层，自行实施。
**问题:** 架构决策未经审查，可能与系统设计冲突。
**正确做法:** 遇到架构级变更时暂停执行，记录发现和建议，等待用户决策。

## 恢复反模式

### 1. 无限重试循环

**现象:** 验证失败 → 生成修复计划 → 再次失败 → 再次生成修复计划...
**问题:** 消耗大量 context，越修越偏离正确方向。
**正确做法:** 最多执行 1 次 gap closure。如果修复后仍然失败，暂停并报告给用户。

### 2. 跳过失败阶段

**现象:** 阶段验证失败后直接进入下一阶段。
**问题:** 后续阶段依赖前序阶段的产出，基础不牢导致连环失败。
**正确做法:** 阶段验证必须通过（或有明确的 Override 理由）才能推进。

### 3. 修改已有 Commit

**现象:** 用 `git commit --amend` 修改之前的提交。
**问题:** 可能覆盖其他 agent 的工作，丢失历史记录。
**正确做法:** 始终创建新的 commit。如果需要修正，创建一个新的 fix commit。
