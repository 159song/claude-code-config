   - Quoted strings (single or double) -> strip quotes
   - `/^\d+$/` -> parseInt

2. Rewrite `parseFrontmatter(content)`:
   - Keep the `---\n` / `\n---\n` boundary detection (no change)
   - Replace the line loop with nested-aware parsing:
     - Track `currentParent = null`
     - If line matches `/^(\s+)([\w][\w_]*):\s*(.*)$/` AND currentParent is set: add to `frontmatter[currentParent]` sub-object using parseYamlValue
     - If line matches `/^([\w][\w_]*):\s*(.*)$/` (top-level): if value is empty string, set `frontmatter[key] = null` and `currentParent = key`; otherwise set `frontmatter[key] = parseYamlValue(value)` and `currentParent = null`
     - Skip empty lines (reset currentParent only if truly empty, not if just whitespace-only)
   - After loop: for any key still null that was set as currentParent but got sub-keys, the sub-object is already in place (the null gets overwritten when first sub-key is encountered)

3. Add `serializeFrontmatter(fm)` function:
   - For each key/value in frontmatter object:
     - If value is non-null object (not array): write `key:` on its own line, then each sub-key as `  subKey: formatYamlValue(subVal)`
     - Otherwise: write `key: formatYamlValue(val)`
   - Add `formatYamlValue(val)`:
     - null/undefined -> `'null'`
     - boolean/number -> `String(val)`
     - string containing `/[:#\[\]{}|>!&*?,]/` or matching `/^\d{4}-\d{2}/` -> wrap in double quotes
     - Otherwise -> bare string

   - If key contains `.`, split on first `.` to get `[parentKey, childKey]`

   - Set `frontmatter[parentKey][childKey] = parseYamlValue(value)`
   - Use serializeFrontmatter to write back (full re-serialize for nested mutations)

6. Update `module.exports` to export: `parseFrontmatter, serializeFrontmatter, parseYamlValue, parseStateMd, stateGet, stateSet, stateJson, run`

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI args -> state.cjs | User-provided keys and values from command line |
| JSON string -> stateMerge | User-provided JSON parsed and merged into frontmatter |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | statePatch key names | mitigate | Key names from `--key` args are used as frontmatter keys directly. Validate key matches `/^[\w][\w_.]*$/` pattern before accepting. Reject keys containing path separators or special chars. |
| T-02-02 | Tampering | stateMerge JSON input | mitigate | Wrap JSON.parse in try/catch, exit with error on invalid JSON. Only accept plain objects (not arrays) at top level. |
| T-02-03 | Tampering | Dotted key path traversal | mitigate | Split dotted key on first `.` only (max 2 levels). Do not allow arbitrary depth traversal. Validate each segment matches `/^[\w][\w_]*$/`. |
| T-02-04 | Information Disclosure | stateValidate output | accept | Validation output reveals frontmatter structure, but STATE.md is local project file with no secrets. Low risk. |
</threat_model>
