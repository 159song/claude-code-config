    - description: 审查源代码质量、安全性、性能问题，生成结构化 REVIEW.md

    Input contract section using `<input_contract>` XML block (per D-11):
    Required fields table: phase (number), files (filepath[]), review_path (filepath), depth (string: quick|standard|deep)
    Optional fields table: diff_base (string), config (object)

    Content sections:
    - **角色**: Code reviewer that reads source files and produces structured REVIEW.md
    - **审查维度** (per D-12): bugs, security vulnerabilities, code quality, performance issues
    - **深度控制** (per D-03): `quick` = surface scan (naming, obvious issues), `standard` = full review (all 4 dimensions), `deep` = exhaustive (line-by-line, cross-file analysis)
    - **审查流程**: For each file in `files` list: Read file -> analyze against 4 dimensions -> record findings with IDs (CR-01, CR-02, etc.) and severity (critical/high/medium/low) -> accumulate
    - **REVIEW.md 输出格式**: YAML frontmatter with `status: clean|issues_found|error`, `depth`, `phase`, `files_reviewed`, `findings: { critical, high, medium, low, total }`, `iteration`. Body contains `## Findings` with per-finding sections: ID, severity, file, line range, description, suggested fix.
    - **文件数量保护** (from Research pitfall 4): If files.length > 50, auto-downgrade to `quick` depth and warn in REVIEW.md frontmatter.
    - **Finding ID 持久性** (from Research pitfall 1): Use `CR-{NN}` format. On iteration > 1, read previous REVIEW.md to reuse IDs for same-file/same-issue findings.

    Output contract section using `<output_contract>` XML block:
    Artifacts table: REVIEW.md (required)
    Completion marker JSON: { "status": "complete|partial|failed", "artifacts": ["<review_path>"], "summary": "..." }

    **Create `wf/workflows/code-review.md`** (per D-01, D-04, D-05, D-06, D-13):

    Purpose block: Review source files changed during a phase, auto-fix issues, iterate up to 3 rounds.

    Steps:

    1. `<step name="initialize">`: Parse arguments ($ARGUMENTS). Extract phase number, --depth override, --files override. Run `wf-tools init phase-op <phase>` for phase_dir, padded_phase. Load config via `wf-tools config` to get `workflow.code_review`, `workflow.code_review_depth`, `workflow.code_review_auto_fix`, `workflow.code_review_max_iterations`, `agents.models.reviewer`.

    2. `<step name="check_config_gate">`: If `config.workflow.code_review === false`, display "Code review disabled in config" and exit.

    3. `<step name="compute_file_scope">`: Run `wf-tools review scope <phase> [--files ...]` to get file list and tier. If files empty, warn user and suggest `--files` flag, exit. Display file count and tier.

    4. `<step name="review_fix_chain">`: The core Skill() chain loop (per D-04, D-06):

       iteration = 0
       max_iterations = config.workflow.code_review_max_iterations (default 3)
       review_path = <phaseDir>/REVIEW.md

       LOOP (max iterations):
         iteration++
         Step A: Skill(wf-reviewer) with phase, files, review_path, depth
         Step B: Run `wf-tools review parse <review_path>` to get findings
         Step C: IF status === 'clean' OR findings.total === 0 -> BREAK
         Step D: IF config.workflow.code_review_auto_fix -> construct fix tasks from findings, Skill(wf-executor) to apply fixes
         Step E: Loop back for re-review
       END LOOP

    5. `<step name="present_results">`: Display final REVIEW.md summary using ui-brand.md formatting. Show findings count, fixed count, remaining issues. If max iterations reached with remaining issues, clearly list them.

    6. `<step name="commit_review">`: Git commit REVIEW.md via `wf-tools commit --message "docs(phase-{N}): code review" --files <review_path>`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User args -> file scope | --files and --depth args passed to review workflow |
| Reviewer agent -> REVIEW.md | Agent-generated content consumed by fix chain |
| Fix chain -> source code | Auto-fix modifies production source files |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-07 | Tampering | code-review.md --depth arg | mitigate | Validate depth is one of ['quick', 'standard', 'deep'] in initialize step; reject invalid values |
| T-06-08 | Tampering | code-review.md --files arg | mitigate | Delegated to review.cjs computeFileScope which validates paths within cwd (T-06-01) |
| T-06-09 | Information Disclosure | wf-reviewer agent file reading | mitigate | Review scope filtered by filterReviewFiles which excludes .env, credentials, secrets |
| T-06-10 | Elevation of Privilege | Auto-fix via wf-executor | accept | Executor runs in worktree isolation; fixes are git-committed (reversible); user reviews during UAT |
</threat_model>
